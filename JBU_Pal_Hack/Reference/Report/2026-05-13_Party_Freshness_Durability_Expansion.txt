[1] 작업 목표
───────────────────────────────────────────────────────────────────────────────
직전 2026-05-12 리포트(Pal Storage 변조 + 탭 UI) 후속으로 다음 세 가지를 추가
한다.

  (A) 보유 팰 변조의 적용 범위를 Pal Box 외에 "데려다니는 팰(Otomo / Party)"
      까지 확장. SDK 헬퍼를 컨테이너 베이스 인자를 받는 형태로 일반화한 뒤,
      UI 에서 Source(Box / Party) 라디오로 전환.

  (B) 인벤토리 음식 부패 정지. UPalItemSlot 의 CorruptionProgressValue 를
      매 프레임 0 으로 강제 → 영원히 신선. 단일 슬롯 / 컨테이너 일괄 두 가지
      모드 제공.

  (C) 장비 내구도 / 무기 잔탄 강제. UPalDynamicArmorItemDataBase /
      UPalDynamicWeaponItemDataBase 인스턴스에 직접 R/W. TWeakObjectPtr 해소
      는 현 단계에서 보류, 사용자가 한 번 hex 주소를 잡아 입력하는 패턴으로
      bootstrap.

[2-1] Otomo(데려다니는 팰) 컨테이너 체인

  UPalOtomoHolderComponentBase  : UActorComponent
    +0x110  UPalIndividualCharacterContainer*  CharacterContainer
            ↑ Pal Box(PalStorage.TargetContainer)와 완전히 동일한 타입.
              SlotArray@0x80, Slot.ReplicateIndividualParameter@0x98 동일
              → 기존 IndividualParam:: 오프셋 테이블 100% 재사용 가능.

  ※ 단, UPalOtomoHolderComponentBase 는 ActorComponent 라 SDK 덤프 전체에서
    "OtomoHolder* 멤버 필드"가 존재하지 않음. 런타임에 부착되는 컴포넌트라
    정적 오프셋 한 줄로 도달하는 경로가 없음.

  현 단계 해결책 = 사용자 입력 (수동 hex 주소).
  중기 해결책   = AOB 후크로 OtomoHolderComponent 멤버 함수 진입 시 this 캐치.


[2-2] 보유 팰 일반화 ─ "어떤 UPalIndividualCharacterContainer 든 동일 코드"

    <Container 베이스>
      → SlotArray.Data (+0x80) / SlotArray.Num (+0x88)
        → Slot[i]
          → ReplicateIndividualParameter (+0x98)
            → SaveParameter base = +0x388
              → 기존 IndividualParam:: 오프셋(Level/Exp/HP/MP/Sanity/Talents...)
                전부 그대로 재사용

  → 모든 GetPal*/SetPal* 를 *In(uintptr_t container, int slot) 시그니처로
    재정렬. 기존 Pal Box 전용 함수는 wrapper 로 살아남음.


[2-3] UPalItemSlot 추가 멤버 (음식 신선도)

  UPalItemSlot
    +0x154  int32  StackCount               (기존)
    +0x158  float  CorruptionProgressValue  ← 신규
            (0.0 = 신선, 1.0 = 부패 완료. RepNotify 멤버이며 클라이언트에서
             직접 0 으로 매 프레임 덮어쓰면 부패 진행이 정지된 것처럼 동작.)
    +0x190  TWeakObjectPtr<UPalDynamicItemDataBase>  DynamicItemData
            (이 멤버는 자동 해소가 어려워 [2-4] 별도 경로로 우회.)


[2-4] 장비 Dynamic Item Data 레이아웃 (Armor / Weapon 공통)

  UPalDynamicArmorItemDataBase  : UPalDynamicItemDataBase
    +0x78  float  Durability
    +0x7C  float  MaxDurability

  UPalDynamicWeaponItemDataBase : UPalDynamicItemDataBase
    +0x78  float  Durability
    +0x7C  float  MaxDurability
    +0x84  int32  RemainingBullets

  → Durability / MaxDurability 오프셋이 두 클래스에서 완전히 동일하므로
    별도 분기 없이 같은 DynamicItemData:: 네임스페이스로 통합. RemainingBullets
    는 Weapon 전용이지만 Armor 객체에 같은 오프셋으로 쓰면 무해(읽기 0).


[2-5] TWeakObjectPtr 해소가 어려운 이유 (자동 도달 보류 사유)

  UPalItemSlot.DynamicItemData 는 raw 포인터가 아니라
      struct FWeakObjectPtr {
          int32 ObjectIndex;        // +0x00
          int32 ObjectSerialNumber; // +0x04
      };
  로, 객체 주소를 얻으려면 GUObjectArray (FChunkedFixedUObjectArray) 를 통한
  chunk lookup 이 필요하다.
      chunkIndex = ObjectIndex / 65536
      itemIndex  = ObjectIndex % 65536
      FUObjectItem* chunk = ((FUObjectItem**)(GUObjectArray + ObjObjectsOffset))[chunkIndex]
      UObject* obj = *(UObject**)(chunk + itemIndex * 24)

  현재 Offsets.h 에는 GUObjectArray 가 등록돼 있지 않고, 빌드마다 위치가
  바뀌어 AOB 스캔 모듈이 별도로 필요함. 이번 작업에서는 채택하지 않고
  Equipment Durability 는 사용자 hex 입력 방식으로 대체.

[4] 안전성/안티치트 고려
───────────────────────────────────────────────────────────────────────────────
  - 신규 모든 setter 는 기존 WriteAt<T>() 흐름(IsBadWritePtr 가드 후 raw write)
    그대로 사용. 추가적인 메모리 권한 변경 없음.

  - Otomo 컨테이너 / Dynamic Item Data 객체 주소는 사용자가 잘못 입력하면
    IsBadWritePtr 가 false 를 반환해 setter 가 무사 실패 → 크래시 회피.
    단, 게임이 해당 객체를 해제(파기)한 뒤에도 주소가 유효해 보이는 경우
    (use-after-free) 가 이론적으로 가능하므로, UI 라이브 표시값이 명백히
    이상하면(Durability 가 NaN 등) 토글을 끄고 주소를 재확보할 것.

  - bFreezeAllFoodFreshness 의 슬롯 순회 상한(64)은 일반 컨테이너 크기에
    여유 있게 설정한 값. 실제로는 SDK 의 SlotArray.Num 을 읽어 동적으로
    잡아도 되지만 매 프레임 호출되는 경로라 고정 상한이 단순/빠름.

  - CorruptionProgressValue 와 Durability 모두 RepNotify 멤버이므로
    서버 권위 모드(공식 서버)에서는 다음 OnRep 시 되돌려질 수 있음.
    솔로/전용 서버 환경에서 의도된 동작이며, 멀티 서버 호환은 Track 2
    함수 후킹 마이그레이션 영역.
