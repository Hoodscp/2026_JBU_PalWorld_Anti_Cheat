[2-1] BuildableTest (Can Build Anywhere)  ─ 조건분기 강제

  CT 원문 sig: 40 38 B1 E4 00 00 00   (7바이트)
    40 = REX                                  (sil 사용 위해 필수)
    38 B1 E4 00 00 00 = cmp byte ptr [rcx+0xE4], sil

  기존 게임 동작: 빌드 가용성 검사. ZF=1 일 때만 통과.
  변조 설계 (MidHook bodyEnabled):
    1) mov byte ptr [rcx+0xE4], sil    ; 메모리를 sil 값으로 강제 동기화 (7B)
    2) cmp byte ptr [rcx+0xE4], sil    ; 원본 재실행 — ZF=1 보장 (7B)
  → 총 14B. patchLen = 원본 7B.

[2-2] CanCatchBossPal  ─ 조건분기 강제

  CT 원문 sig: 48 8B 83 30 06 00 00 80 B8 98 05 00 00 00   (14바이트)
    48 8B 83 30 06 00 00 = mov rax, [rbx+0x630]          (보스 데이터 ptr)
    80 B8 98 05 00 00 00 = cmp byte ptr [rax+0x598], 0   (보스 차단 플래그, imm8=0)

  기존 게임 동작: [rax+0x598] 가 0 이 아니면 페일 스피어로 보스/머천트 포획 차단.
  변조 설계 (MidHook bodyEnabled):
    1) mov rax, [rbx+0x630]                   ; 원본 (7B)
    2) mov byte ptr [rax+0x598], 0            ; ★ 차단 플래그 0 강제 주입 (C6 80 ... 00, 8B)
    3) cmp byte ptr [rax+0x598], 0            ; 원본 재실행 — 이제 항상 ZF=1 (7B)
  → 총 22B. jeOffsetToDisabled = 22+5 = 27 ≤ 127 (MidHook je rel8 한계 통과).

[2-3] PalStatsCreation (Perfect IV)  ─ 레지스터 강제값 주입

  CT 원문 sig (uniqueness 포함): 41 BE 65 00 00 00 44 2B F7 45 85 F6 0F  (13B)
  패치 사이트(원본 6바이트, 첫 단일 명령어):
    41 BE 65 00 00 00 = mov r14d, 0x65
    (이후 명령어 44 2B F7 = sub r14d, edi 가 edi 를 IV 산정 기준값으로 사용)

  기존 게임 동작: edi 에 들어있는 정상 분포값(50 등) 기준으로 IV 계산.
  변조 설계 (MidHook bodyEnabled):
    1) mov edi, 0x64           ; ★ 강제 주입 — IV = 100 (BF 64 00 00 00, 5B)
    2) mov r14d, 0x65          ; 원본 재실행 (6B)
  → 총 11B. 직후의 sub r14d, edi 가 edi=0x64 로 평가되어
    HP/근접/원거리/방어 IV 모두 100.
  적용 범위: 새로 생성/포획되는 펄 한정. 이미 잡힌 펄은 기존 SetPalTalents
  (Track 1) 가 담당.

[2-4] ResearchIncrease (Quick Research)  ─ 명령어 추가 주입

  CT 원문 sig: F3 0F 58 77 14   (5바이트, 정확히 1 명령어)
    F3 0F 58 77 14 = addss xmm6, [rdi+0x14]   (연구 진행도 누적)

  기존 게임 동작: 매 tick 작은 float 양만큼 선형 누적.
  변조 설계 (MidHook bodyEnabled):
    1) addss xmm6, [rdi+0x14]   ; 원본 재실행 (5B)
    2) mulss xmm6, xmm6         ; ★ 제곱 누적 주입 (F3 0F 59 F6, 4B)
  → 총 9B. patchLen = 5B (정확히 jmp 공간).
  진행도가 (x + δ)² → x² + 2xδ + δ² 패턴으로 폭증해 몇 호출이면 한 연구 완료.

[4] 헬퍼 인터페이스 활용 — 새 저수준 코드 0줄
───────────────────────────────────────────────────────────────────────────────
신규 4개 모듈 모두 기존 헬퍼 그대로 사용:

  Memory/MidHook.{h,cpp}   ← bodyEnabled/bodyDisabled 양쪽 분기 트램폴린
  SDK/Signatures.h         ← AOB 문자열
  GUI/Menu.h               ← 토글 bool

신규 vendoring·신규 트램폴린 생성기·신규 NopHook 변종 모두 불필요. 이는
MidHook 의 일반화 설계(2026-05-13 보고서 §3 헬퍼화)가 의도한 대로
"한 후킹 = 한 파일 = .h 3함수 + .cpp 약 30줄" 비용으로 신규 후크 추가가
정착되었음을 보여준다.


[5] 안전성 / 안티치트 고려
───────────────────────────────────────────────────────────────────────────────
  - bodyEnabled 안에 RIP-relative / 짧은 점프 0개. (모두 mov/cmp 변형 + 상수
    imm8/imm32 — 위치 의존 명령어 아님.) 트램폴린 재배치 후에도 안전.
  - 모든 후크는 sig 빈 문자열 / 패턴 미발견 시 Install 이 nullptr 반환 →
    Tick 의 SetEnabled 도 no-op (MidHook::SetEnabled 가 null guard) →
    DLL 만 인젝션하고 toggle 만 켜도 크래시하지 않음.
  - Shutdown 은 HookManager::Shutdown 이전에 호출되도록 HackMain 에 명시 배치.
    누락 시 DLL 언로드 후 dangling jmp 위험은 기존 P0/P1 후크와 동일.
  - CanCatchBoss: `[rax+0x598]` 메모리를 0 으로 write 함. 이 데이터 객체는
    포획 시도 시점에 한 번만 평가되므로, 토글 OFF 후 다음 보스 진입 시
    원래 값(보호 플래그)으로 자연 복구된다.
  - PalStatsCreation: edi 가 IV 산정의 한 입력일 뿐, 다른 입력(레벨/종족
    base) 까지 100 으로 만들지는 못함. CT 의 §B 패턴 그대로의 효과 범위.
  - ResearchIncrease: float 누적이라 큰 값에서 부동소수 정밀도 손실 가능 →
    한 번 완료 후엔 자동으로 다음 연구로 전환되니 무해.
  - 본 4종은 모두 게임 실행 코드 .text 를 바이트 단위로 변조 → External
    의 IntegrityScanner 가 정확히 잡는다. 안티치트 측에선 본 변조도
    "Track 2 인라인 후크" 카테고리로 동일하게 가시화됨.
