# 난독화 및 보호 분석 (Obfuscation and Protection Analysis)

## 1. 동적 API 분석 (Dynamic API Resolution)
백신 프로그램의 탐지나 단순 정적 분석을 피하기 위해, 트레이너는 주요 기능들을 직접 임포트하지 않습니다. 대신:
- 실행 시점에 `kernel32.dll` 또는 `user32.dll`을 로드합니다.
- `GetProcAddress`를 사용하여 `WriteProcessMemory`, `ReadProcessMemory`, `OpenProcess`와 같은 함수의 주소를 메모리에서 직접 찾습니다.
- 이를 통해 실행 파일의 임포트 주소 테이블(IAT)에서 트레이너의 실제 능력을 숨깁니다.

## 2. 리소스 난독화
트레이너의 핵심 로직과 GUI는 `REMOTE`라는 이름의 임베디드 리소스로 저장되어 있습니다.
- 이 리소스는 일반적인 파일 형태가 아니며, 압축되거나 변형된 상태로 존재할 가능성이 높습니다.
- 실행 시에만 임시 폴더로 복호화 및 추출됩니다.

## 3. 무작위 파일명 사용
트레이너는 추출된 구성 요소(예: `tr_speedhack_randomString.dll`)에 대해 무작위 파일명을 생성합니다. 이는:
- 안티 치트 소프트웨어가 단순한 파일 경로 기반 탐지를 수행하기 어렵게 만듭니다.
- 트레이너의 여러 버전이 실행될 때 파일 잠금(Locking) 문제가 발생하는 것을 방지합니다.

## 4. .NET 호스팅
자체 .NET 런타임을 호스팅함으로써 트레이너는 다음과 같은 이점을 얻습니다:
- 외형은 "네이티브" EXE처럼 보이면서도 GUI와 복잡한 로직에는 C# 등 고수준 언어를 사용할 수 있습니다.
- 추출된 어셈블리에 .NET 전용 난독화 기술을 적용할 수 있습니다.

## 5. 치트 엔진 스크립트 활용
하드코딩된 메모리 주소 대신 AOB (Array of Bytes) 스캔을 사용하는 것은 게임 업데이트에 대한 일종의 보호책입니다. 게임 실행 파일이 재컴파일되어 주소가 바뀌더라도 트레이너가 계속 작동하도록 보장합니다.

## 6. 실행 흐름 난독화
바이너리는 상태 설정을 위해 작고 중복되어 보이는 수많은 함수와 정적 초기화 루틴을 사용합니다. 이는 함수 개수를 부풀려 분석가가 "메인" 로직을 찾는 것을 지루하고 어렵게 만듭니다.
- 광범위한 구조적 예외 처리(SEH) 사용 또한 디버깅과 트레이싱을 복잡하게 만드는 역할을 합니다.
